## 改善後の型定義案

### 施設基本情報

- `Spot`
- 施設名、エリア、初心者向け度、要約、アクセスメモ、特徴、安全メモを持つ

### 施設ルール情報

- `FacilityRule`
- タイトル、説明、重要度を分けて持つ

### 日別予測情報

- `DailySpotForecast`
- 釣り場ID、日付、魚候補、時間帯、潮、海況、最近の様子、回遊期待、理由文、注意文をまとめる

### 魚種候補

- `SpeciesCandidate`
- 魚名、期待度、おすすめの始め方、短い説明、データ状態を持つ

### 理由文

- `ReasonItem`
- タイトル、説明、行動のヒント、参照した情報種別、データ状態を持つ

### 注意文

- `AdvisoryItem`
- レベル、タイトル、説明、行動のヒントを持つ

### 潮情報

- `TideInfo`
- 潮名、満潮干潮、海が動く時間、全体説明、データ状態を持つ

### 海況情報

- `MarineCondition`
- 天気、風、波、水温、全体説明、初心者向けコメント、データ状態を持つ

### 回遊期待度

- `MigrationExpectation`
- 期待度、説明、参考にした近隣スポット、行動ヒント、データ状態を持つ

### 最近の様子

- `CatchSummary`
- データ状態、fallback方針、全体説明、観測項目一覧を持つ

## 欠損データ時の fallback 設計

### データ欠損時の扱い

- `sourceStatus` を `available` `limited` `missing` で表す
- 直近釣果が弱い地点は `CatchSummary` を `limited` にする
- UIは数値の欠損を `未取得` と表示し、空配列にはしない

### 代替表示文言

- 直近釣果が少ない地点: 「この地点だけでは新しい釣果が少ないため、近くの釣り場の様子も合わせて見ています」
- 海況の一部が未取得: 「一部の海況データがないため、無理をしない前提で見ています」
- 回遊情報が弱い: 「回ってくる期待は見ていますが、今日は控えめに考えるのが安心です」

### スコアリング上の fallback 方針

- 直近釣果が `missing`: 高評価にも低評価にも寄せず中立点
- 直近釣果が `limited`: 近隣釣果を使うが上限を低めに抑える
- 海況が欠損: 安全側に倒した控えめな点数
- 回遊情報が弱い: 小さめの加点に留める
- fallbackを使った場合は `ScoreBreakdown.fallbackNotes` に残す

## Repository 整理方針

- `ForecastRepository` は UI が使う最小の読み取り操作だけを持つ
- 日別一覧取得と単地点取得を分ける
- `getRepositoryMeta()` で mock / live の識別を持てるようにする
- UIは `forecastService` だけを参照し、取得元の差し替えは `setForecastRepository()` で行う
